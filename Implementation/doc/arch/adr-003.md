# ADR 3: Web-Based Editor Over VS Code Extension

## Status

Accepted (supersedes initial proposal)

## Context

The original project proposal described SyncSpace as a VS Code extension, inspired by VS Code Live Share. Building a VS Code extension would provide deep integration with a professional IDE, but it introduces significant development complexity.

VS Code extensions are written in TypeScript and run inside the VS Code extension host, which has a restricted API surface. Integrating a CRDT engine into the extension host, hooking into the Monaco editor's document model, and managing WebSocket connections within VS Code's process model requires specialized knowledge of the Extension API. The development-debug cycle for extensions is also slower — each change requires reloading the extension host window.

A web-based approach, on the other hand, uses standard HTML, CSS, and JavaScript running in a browser. Open-source code editor components like CodeMirror 6 provide a full-featured editing experience (syntax highlighting, multiple language modes, customizable keybindings) and have first-class Yjs integration via the `y-codemirror.next` binding. The entire application can be developed and tested by simply refreshing a browser tab.

Our team assessed the effort required for both approaches. The VS Code extension path would consume a large portion of the semester on extension-specific plumbing — activation events, command registration, webview panel lifecycle, extension host API constraints, and the slower debug cycle of reloading the extension host window on every change. These are significant sources of complexity that do not advance the core architectural learning goals (CRDT-based collaboration, stateless relay design). The web-based approach lets us focus almost entirely on the CRDT synchronization and user experience.

## Decision

We will build SyncSpace as a web-based collaborative code editor using CodeMirror 6 as the editor component, rather than a VS Code extension. The application will run in any modern browser. The relay server is implemented in **Python using FastAPI** (see ADR 5).

## Consequences

This decision dramatically simplifies development. We can use standard web development tools (browser DevTools, hot reload) and avoid VS Code Extension API complexity entirely. The `y-codemirror.next` binding provides direct integration between Yjs and CodeMirror, meaning cursor tracking and document synchronization work with minimal glue code.

The trade-off is that we lose the deep IDE integration that a VS Code extension would provide — features like shared debugging, terminal sharing, and file explorer synchronization are no longer feasible. The system focuses on **collaborative editing with syntax highlighting** rather than full IDE functionality; features like compilation, execution, and LSP integration are intentionally out of scope. However, the lost features were "Should" and "Could" priority requirements, not core requirements. The architecturally significant requirements (local-first editing, CRDT conflict resolution, live cursors, WebSocket sync) are fully achievable in a web-based editor.

Browser-based execution introduces measurement variability due to JavaScript event loop scheduling, garbage collection pauses, and rendering pipeline overhead. Performance measurements (particularly NFR-P-02, keystroke latency ≤ 16 ms) will be conducted under controlled conditions using `performance.now()` instrumentation to account for this variability (see SRS § 3.6).

Users will access SyncSpace through a browser URL rather than installing an extension, which is arguably simpler from a usability perspective.
