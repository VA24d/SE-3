# ADR 1: Use CRDT (Yjs) Over Operational Transformation for Conflict Resolution

## Status

Accepted

## Context

SyncSpace requires a mechanism to resolve conflicting edits when multiple users type in the same document simultaneously. There are two established approaches in the collaborative editing literature: Operational Transformation (OT) and Conflict-free Replicated Data Types (CRDTs).

Operational Transformation, used by Google Docs, requires a central server to sequence all operations. Each client sends its edits to the server, which transforms them against concurrent operations before broadcasting the result. This approach works well at scale but creates a single point of failure and makes offline editing difficult — the server must be available to correctly order operations.

CRDTs take a fundamentally different approach. Each client maintains a complete local replica of the document, and edits are applied locally without waiting for the server. The CRDT data structure is mathematically designed so that replicas always converge to the same state regardless of the order in which operations are received. This eliminates the need for a central sequencer.

Our project has three key constraints: we are a small team building a prototype within a semester, we want to support temporary offline editing, and we want to avoid a complex central server. Additionally, mature open-source CRDT libraries exist — notably Yjs, which provides sub-millisecond local merge performance under typical small-document workloads (as reported in Yjs benchmarks) and a built-in awareness protocol for cursor sharing.

## Decision

We will use Yjs as our CRDT engine for document synchronization. Each client will maintain a Yjs `Y.Doc` instance as the local-first replica. The relay server will not interpret or transform operations — it will only forward binary CRDT update vectors between clients.

## Consequences

On the positive side, this decision gives us mathematically guaranteed **syntactic convergence** — all replicas that receive the same set of updates will reach an identical document state, regardless of delivery order. However, CRDTs do not guarantee **semantic correctness**: concurrent edits to overlapping regions may merge into unintended output (e.g. interleaved code blocks), and users must still review the merged result. The decision also enables a local-first editing experience where keystrokes are never blocked by network latency, and it makes the relay server extremely simple since it is just a message forwarder.

On the negative side, CRDT memory usage grows with the **number of operations** (inserts, deletes, and causal metadata/tombstones), not just the final document size. A document whose final text is 100 KB may consume significantly more memory internally if it has undergone heavy editing (repeated typing and deletion). Given our prototype's short-lived sessions and small expected edit volumes, we expect this overhead to remain manageable. For long-running or heavily-edited sessions, Yjs provides garbage-collection mechanisms (`gc` option) that can reclaim tombstone memory.

Another consequence is that we are now dependent on the Yjs library. If Yjs introduces breaking changes or becomes unmaintained, we would need to migrate to an alternative like Automerge. Given that Yjs is actively maintained and widely adopted, we consider this risk acceptable.

**Architectural Dependencies:** This decision constrains the communication layer (ADR-002): the relay must transport opaque binary Yjs update vectors and multiplex the Yjs Awareness protocol alongside document updates. Any change to the CRDT library would require corresponding changes to the wire protocol and WebSocket provider.
