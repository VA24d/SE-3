# ADR 5: Python and FastAPI for the Relay Service

## Status

Accepted

## Context

The relay server must accept WebSocket connections, optionally serve the static web client, and broadcast opaque CRDT frames between peers. A natural implementation path in the Yjs ecosystem is **Node.js** with the **`ws`** package and helpers such as **`y-websocket`**, because many tutorials assume that stack.

Our team also considered **Python** with an **ASGI** framework. Python is widely used in teaching and DevOps scripts, integrates cleanly with **`venv`**, and **FastAPI** exposes WebSocket endpoints with minimal boilerplate while remaining easy to read for graders and teammates. The client already uses a **small custom WebSocket provider** (framed Yjs and awareness messages) rather than a large server-side Yjs binding, so we are not required to run Yjs on the server for correctness.

The forces in tension are **ecosystem familiarity for CRDT demos** (Node) versus **simplicity of the Python runtime**, **consistent tooling with other course work**, and **keeping the server as a thin forwarder** that does not need a JavaScript CRDT runtime at all.

## Decision

We will implement the relay as **Python 3** with **FastAPI** and **Uvicorn**, using Starlette’s WebSocket support for session-scoped broadcast. The server will remain a **message forwarder** (see ADR 4); it does not interpret CRDT payloads. Static files for the browser client are mounted from the same process for local demos.

## Consequences

On the positive side, deployment is a single **Python virtual environment** and **`pip install -r requirements.txt`**, without managing a separate Node runtime for the relay. The codebase stays small and readable, and WebSocket routes are explicit in one module.

On the negative side, we do not use **`y-websocket`**’s reference server, so any future feature that assumes that server (auth hooks, shared persistence plugins tied to Node) would require extra work. If we later need parity with the Node ecosystem, we could re-implement the relay in Node while keeping the same wire protocol, or extract the protocol into a shared specification.

This decision does not change ADR 2’s choice of **WebSocket as the transport**; it only selects the **implementation language** for the relay process.
