# ADR 4: Stateless Relay Server Over Centralized Document Authority

## Status

Accepted — deployment and stack details in Consequences amended by [ADR 8](adr-008.md).

## Context

In a collaborative editing system, the server can play one of two roles. In a centralized authority model (as used by Google Docs with OT), the server is the single source of truth for the document. All edits pass through the server, which transforms and orders them before distributing the canonical result. The server stores the full document and its edit history, and clients are essentially thin views of the server's state.

In a relay model, the server is a simple message forwarder. It receives binary CRDT update vectors from one client and broadcasts them to all other clients in the same session. The server does not parse, interpret, or transform the content of these messages. Each client's CRDT replica is a fully autonomous copy of the document, and correctness is guaranteed by the CRDT's mathematical properties, not by the server.

Our choice of Yjs as the CRDT engine (see ADR 1) already implies that conflict resolution happens at the client level. The question is whether the relay server should additionally maintain a persistent copy of the document for durability, or whether it should be kept minimal.

For our prototype, we prioritize simplicity. A stateless relay that only holds the latest CRDT state in memory (for new-joiner synchronization) and discards it when all clients disconnect is the simplest possible server architecture. It requires no database, no persistent storage, and no backup strategy.

## Decision

We will implement the relay server as a stateless WebSocket message router. It will maintain an in-memory copy of the Yjs document state per active session (for initial sync when a new client joins), but it will not persist document state to disk or a database. When all clients leave a session, the session state is discarded after a short timeout.

## Consequences

This decision makes the relay server trivially simple to implement and deploy. It is a **Python** service (FastAPI + Uvicorn) in a small number of modules, requiring no database setup, no migration scripts, and no backup infrastructure for the MVP. Deployment is a virtual environment, `pip install -r requirements.txt`, and starting the ASGI app (see README).

The main negative consequence is that documents are not persisted. If all clients disconnect and the session times out, the document is lost. For our prototype, this is acceptable — sessions are ephemeral collaboration windows, not long-term storage. If persistence were needed in the future, we could add a persistence layer to the relay (Yjs supports LevelDB-based persistence via `y-leveldb`) without changing the client architecture at all.

Another consequence is that the relay is a single point of availability — if it crashes, no messages are routed until it restarts. However, because each client has a full local CRDT replica, no data is lost during a relay outage. Clients simply buffer operations locally and resync when the relay returns. This aligns with our fault tolerance requirement (NFR-R-02).
