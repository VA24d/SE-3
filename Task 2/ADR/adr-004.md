# ADR 4: Stateless Relay Server Over Centralized Document Authority

## Status

Accepted — deployment and stack details in Consequences amended by [ADR 8](adr-008.md).

## Context

In a collaborative editing system, the server can play one of two roles. In a centralized authority model (as used by Google Docs with OT), the server is the single source of truth for the document. All edits pass through the server, which transforms and orders them before distributing the canonical result. The server stores the full document and its edit history, and clients are essentially thin views of the server's state.

In a relay model, the server is a simple message forwarder. It receives binary CRDT update vectors from one client and broadcasts them to all other clients in the same session. The server does not parse, interpret, or transform the content of these messages. Each client's CRDT replica is a fully autonomous copy of the document, and correctness is guaranteed by the CRDT's mathematical properties, not by the server.

Our choice of Yjs as the CRDT engine (see ADR 1) already implies that conflict resolution happens at the client level. The question is whether the relay server should additionally maintain a persistent copy of the document for durability, or whether it should be kept minimal.

For our prototype, we prioritize simplicity. A stateless relay that acts as a pure blind router — forwarding opaque binary frames without parsing or interpreting their CRDT content — is the simplest possible server architecture. It requires no CRDT engine on the server, no database, no persistent storage, and no backup strategy. The server holds only an in-memory set of active WebSocket connections per session, not the Yjs document state itself.

## Decision

We will implement the relay server as a **pure blind WebSocket message router**. The relay maintains only a **set of active WebSocket connections per session** — it does not maintain, parse, or compact Yjs document state. When a new client joins a session that already has connected peers, the relay forwards a JSON `request_state` control message to an existing peer, which responds with its full `Y.Doc` state encoded via `Y.encodeStateAsUpdate()`. The relay forwards this response to the joiner as an opaque binary frame. This **peer-driven catch-up** model keeps the server completely unaware of the CRDT payload semantics.

When all clients leave a session, the session entry (the set of WebSocket references) is removed. The relay does not persist any document data to disk or a database.

## Consequences

This decision makes the relay server trivially simple to implement and deploy. It is a **Python** service (FastAPI + Uvicorn) in a small number of modules, requiring no CRDT library on the server, no database setup, no migration scripts, and no backup infrastructure for the MVP. Deployment is a virtual environment, `pip install -r requirements.txt`, and starting the ASGI app (see README).

A key consequence of the peer-driven catch-up design is that **a new client can only join an active session if at least one existing peer is present** to serve the current document state. If no peers are connected, there is no state to recover — the session is empty.

The main negative consequence is that documents are not persisted. If all clients disconnect simultaneously, the document state is irrecoverably lost because no replica remains — neither on the relay (which never held one) nor on any client. This is an accepted trade-off for the prototype's stateless design; sessions are ephemeral collaboration windows, not long-term storage. If persistence were needed in the future, we could add a persistence layer (e.g. a `y-py`-backed compaction service or `y-leveldb` in a Node sidecar) without changing the client architecture at all.

Another consequence is that the relay is a single point of availability — if it crashes, no messages are routed until it restarts. **This can cause permanent data loss.** The prototype's fault-tolerance mechanism for unexpected WebSocket disconnections is a simple timed page reload (see System.md § Communication Subsystem), which destroys the browser's in-memory `Y.Doc` replica. If the relay crashes and a client's reload fires before the relay recovers, that client's unsynced local edits are irrecoverably lost. This is a known and accepted prototype limitation documented in the SRS trade-off table (§ 3.8). A production system would replace the destructive page reload with a non-destructive WebSocket reconnect loop that preserves the local `Y.Doc` instance, enabling true local buffering and lossless resync.
