# ADR 8: Versioned Correction of Relay and Client WebSocket Implementation Details

## Status

Accepted

## Context

When ADR 2 and ADR 4 were first written, they followed the most common Yjs tutorial path: a **`y-websocket` client provider** and a **Node.js reference server** built with `ws` and related helpers. That text accurately described a popular integration style but did not match what we actually built.

ADR 5 already records the decision to implement the relay in **Python 3 with FastAPI**. The browser client uses **Yjs** and **`y-protocols` awareness** on the wire, but connects through a **small custom WebSocket provider** (prefixed binary frames and a JSON `request_state` handshake) so the relay can remain a dumb forwarder without a JavaScript CRDT runtime. Readers comparing early PDFs or printouts of ADR 2 and ADR 4 to the repository might therefore see a mismatch between narrative and code.

We need a **numbered, explicit amendment** so the change is traceable rather than a silent rewrite of older ADRs.

## Decision

We **amend the implementation details** described in ADR 2 (Decision and Consequences) and ADR 4 (Consequences): the authoritative stack is **WebSocket transport** (unchanged from ADR 2), **stateless relay role** (unchanged from ADR 4), **Python / FastAPI relay** (ADR 5), and **custom framed WebSocket client** in the web app. The earlier Node.js / `y-websocket` server wording is **not** the current decision; it remains historically understandable from this ADR and from the updated bodies of ADR 2 and ADR 4 in the repo.

## Consequences

Future developers can follow the ADR chain: **ADR 8 → revised ADR 2 and ADR 4 → ADR 5** for how sync actually works in code. Course staff and teammates can cite ADR 8 when explaining why an older document snapshot mentioned `node server.js`. If we ever migrate the relay to Node or adopt `y-websocket` end to end, that should be a **new** ADR (new number), per project rules, rather than reusing ADR 8.
