# ADR 7: No Database for the MVP Relay (Ephemeral Sessions Only)

## Status

Accepted

## Context

Persistent collaboration systems almost always store documents in a **database** or object store: relational tables for metadata, blob storage for snapshots, or CRDT-specific backends for efficient sync. Options include **PostgreSQL**, **SQLite**, **Redis**, **S3**, or embedded stores such as **LevelDB** with **Yjs** persistence helpers.

Adding persistence would let users return days later to the same URL, support administrative backup, and simplify analytics. It also introduces **schema design**, **migrations**, **backup and restore**, **data retention policies**, and **operational monitoring** — all valuable in production but heavy for a constrained course prototype whose primary artifact is **architecture documentation** plus a **working demo**.

The project's non-functional goals already allow **ephemeral sessions** where losing the document after the last client disconnects is acceptable. The system uses **local-first editing semantics** — edits are applied to the local Yjs `Y.Doc` replica before network sync — but does not provide **durable local persistence** (e.g. IndexedDB or filesystem backup) in this prototype. The open question is whether we should still adopt a lightweight embedded database "for practice."

## Decision

We will **not** introduce a database or durable blob store for the MVP relay. The server stores **no document state** — only an in-memory set of active WebSocket connections per session (consistent with ADR 4's pure blind relay model). When the last peer leaves, the session entry is discarded. Long-term truth for the document lives in **clients' in-memory CRDT replicas** while they are connected; we do not guarantee recovery after total disconnect.

If we need crash resilience or "save my work" later, that will be a **new ADR** and likely a separate persistence service, not an incremental requirement for the prototype.

## Consequences

The relay stays **operationally cheap**: no database provisioning on laptops, no connection strings in configuration, and fewer moving parts for markers to reproduce the project.

The system is designed as an **ephemeral collaborative environment** — sessions are short-lived collaboration windows, not persistent documents. This trades durability for architectural simplicity within the prototype scope.

The main negative consequence is **data loss** when all participants are offline and the server has recycled the session — acceptable for the prototype but unacceptable for a product. Because the prototype uses a **destructive page reload** on WebSocket disconnect (see System.md § Communication Subsystem), clients cannot buffer unsynchronized edits across reconnection attempts, compounding the data-loss risk. A secondary consequence is that we cannot easily implement **server-side search**, **admin dashboards**, or **compliance retention** without revisiting this decision.

This ADR complements ADR 4: ADR 4 explains **why the server does not act as document authority**; ADR 7 explains **why we also omit durable storage** for the MVP.
