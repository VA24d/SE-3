# ADR 2: Use WebSocket for Real-Time Communication Over HTTP Polling

## Status

Accepted — amended by [ADR 8](adr-008.md) (implementation details below supersede the original Node / `y-websocket` server sketch).

## Context

SyncSpace needs a transport mechanism to send CRDT update vectors between clients and the relay server in real time. There are several options available for browser-to-server communication.

HTTP long-polling is the simplest approach: the client repeatedly sends HTTP requests to the server, which holds the connection open until new data is available. This works everywhere but introduces latency overhead from repeated connection setup and HTTP headers on every message. For a system aiming for sub-second edit propagation, this overhead is undesirable.

Server-Sent Events (SSE) provide a server-to-client push channel over HTTP, but they are unidirectional. Our system requires bidirectional communication — clients must both send and receive updates — so SSE alone would require a separate channel for client-to-server messages.

WebSocket (RFC 6455) provides a persistent, full-duplex connection over a single TCP socket after an initial HTTP upgrade handshake. Once established, messages flow in both directions with minimal framing overhead (as low as 2 bytes per frame). Many Yjs tutorials use the `y-websocket` package with a Node reference server; we instead pair Yjs in the browser with a **small custom WebSocket provider** and a **Python / FastAPI** relay so the server stays a thin forwarder without a JavaScript CRDT runtime (see ADR 5 and ADR 8).

WebRTC could enable direct peer-to-peer communication, bypassing the relay server entirely. However, WebRTC connection setup is complex (requiring STUN/TURN servers for NAT traversal), and debugging peer-to-peer mesh networking is difficult for a small team within a semester timeline.

## Decision

We will use WebSocket as the primary transport protocol. The browser runs Yjs and a **minimal custom WebSocket provider** that sends prefixed binary frames for document and awareness updates and uses a small JSON control message for join-time state requests. The relay is implemented in **Python with FastAPI** (ADR 5), broadcasting opaque frames between peers in a session.

## Consequences

WebSocket gives us low-latency, bidirectional communication with minimal per-message overhead, which directly supports our performance requirement of ≤ 1 second end-to-end sync latency. **Merge semantics and CRDT correctness remain entirely in Yjs on the client**; the custom provider only transports opaque Yjs and awareness bytes, so we do not invent a new conflict-resolution protocol while keeping the server simple.

We trade away turnkey features of `y-websocket` (such as its reference reconnection policy and shared assumptions with the Node demo server) in exchange for a relay that matches our course tooling and a single-process demo (see ADR 8).

The main trade-off is that WebSocket connections are persistent and stateful, which means each connected client consumes a socket on the relay server. For our prototype supporting up to 3 concurrent users per session, this is not a concern. At larger scale, the relay server would need horizontal scaling or connection pooling.

A key architectural strength of this design is that Yjs CRDT updates are **idempotent and commutative** — applying the same update twice or in any order produces the same result. This means the system safely tolerates duplicate messages during reconnection or relay retransmission without requiring explicit acknowledgment or deduplication logic at the transport layer.

WebSocket runs over TCP, which may introduce latency spikes due to packet loss and retransmission (**head-of-line blocking**). A lost TCP segment stalls all subsequent frames until the kernel retransmits it. For our prototype's small-scale sessions (≤ 3 users on a LAN), this impact is minimal, but it is a known limitation of any TCP-based real-time protocol.

Both document updates and awareness data (cursors, presence) share a **single multiplexed WebSocket connection** for simplicity, distinguished by a one-byte prefix. This means a burst of high-frequency awareness updates (e.g. rapid cursor movement) could delay delivery of a document update queued behind them. Given the small session size, this trade-off is acceptable; a production system might separate channels or prioritize document frames.

The custom WebSocket provider uses a **simplified page-reload recovery** on unexpected disconnect rather than the in-process exponential-backoff reconnect loop provided by `y-websocket`. This means unsynced local edits may be permanently lost on reload (see ADR 4, System.md § Communication Subsystem). Improving this to a non-destructive reconnect loop is deferred to future work.

Another consequence is that environments with restrictive firewalls or proxies may block WebSocket connections. In practice, modern browsers and networks handle WebSocket well, and for our use case (developer tools in development environments) this is unlikely to be an issue.
