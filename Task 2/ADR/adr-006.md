# ADR 6: Capability URLs for Sessions (No Login) in the Prototype

## Status

Accepted

## Context

Collaboration products usually combine **identity** (who is editing) with **authorization** (who may access a document). Common patterns include corporate single sign-on, email invitations, workspace membership, and role-based access on the server.

SyncSpace is a **semester prototype** aimed at demonstrating **real-time editing architecture**, not at production-grade account management. Requiring OAuth, password storage, or an account database would add significant security review, UI flows, and backend state that do not advance the core learning goals for this course. At the same time, sessions should not be trivially enumerable: a random person should not guess another team’s session ID without the link.

The tension is between **zero-friction joining** (open a URL and type) and **strong access control** (authenticated users only).

## Decision

For the prototype, we will **not** implement user accounts or server-side authorization checks beyond **possession of the session URL**. A session is identified by a **server-generated UUIDv4** (~122 bits of entropy), embedded in the query string as the session ID. This provides a **capability URL** (bearer-token-style access): anyone who possesses the URL may join the session. Awareness metadata (display name, color) is **self-asserted** by the client and not verified by the server.

We treat this as an explicit **capability-URL access model** suitable for trusted classrooms and local networks, not for confidential data.

## Consequences

The positive consequence is **extremely fast onboarding**: share one link and collaborators are editing within seconds, which matches our usability targets and keeps the relay stateless with respect to identity.

The negative consequences are serious in a real deployment: **no audit trail** of who joined, **no revocation** of access without rotating the session ID, **no per-user quotas**, and **no protection** if the link is leaked. Phishing or accidental paste of the link would expose the document.

Because display names are self-asserted and unverified, the system provides **no identity verification or non-repudiation** — any participant may impersonate another by choosing an arbitrary display name. This is an accepted trade-off for the prototype's trusted environment.

Session URLs may be exposed via **browser history, server logs, referrer headers, or screenshots**. Possession of the URL is the sole access credential, so any leakage vector grants full session access. A production system would mitigate this with short-lived tokens, `Referrer-Policy` headers, and `history.replaceState()` to scrub the URL after join.

If SyncSpace were productized, this ADR would be **superseded** by an ADR introducing proper authentication and authorization, likely with server-issued tokens and optional end-to-end encryption for content.

Until then, documentation and demos should **state clearly** that sessions are **not private** against anyone who obtains the URL.
